## Honest reading

**The lift is real but modest.** Real Claude Opus 4.7 with no skill is far more disciplined than I'd assumed when I designed these cases. It catches DST traps, recognizes SCD2 and uses validity-window joins, and treats leakage detection as central to ML prep. On the **leakage** case (04) the skill made literally no difference — the baseline opened with "🚨 Critical Issue: Target Leakage" before being asked.

Where the skill clearly earns its keep:

| Case | What the skill prevented |
| --- | --- |
| 05 (resolve-entities) | Auto-merging Acme despite a domain mismatch; destructive 16 → 8 collapse instead of a non-destructive links mapping |
| 01 (profile-dataset) | Producing a "Cleaned CSV" that silently dropped the duplicate and nullified ambiguous values before the data owner could weigh in |
| 02 (clean-datetime-fields) | Missing the `tz_localize` vs `tz_convert` callout — the single highest-frequency pandas time-zone bug |

The pattern: **the skills add discipline, not knowledge**. The model already *knows* SCD2 joins and leakage. What the skill adds is *restraint* — diagnose first, ask before mutating, preserve originals, document the rule.

## Where the skill barely helped

- Case 03 (validate-join): the model used the SCD2 predicate without prompting. The +1 came from skilled producing an explicit row-count validation block — useful as a regression artifact, but the immediate answer was correct either way.
- Case 04 (detect-data-leakage): tied at 12/12. With this model, on this kind of clean training schema, the skill is redundant.

## What this changes about my recommendation

If you are a strong frontier model user (Opus 4.7, GPT-5-class), the skills' value is concentrated in:

1. **Habit-shaped tasks**: entity resolution, profile-before-clean, "do not mutate without confirmation". These are about discipline, not capability.
2. **Easy-to-miss conventions**: the localize-vs-convert rule, the SCD2 validity predicate format, point-in-time joins for ML.
3. **Ambiguity handling**: the skills consistently push the model to surface decisions to the data owner instead of guessing.

If you are using a weaker model (older versions, smaller distillations, non-Anthropic), the lift is likely larger — these cases would be a fairer test there.

## Where the lift comes from, mapped to failure modes

| Failure pattern | Baseline saw it in | Skill prevents it via |
| --- | --- | --- |
| **Premature cleaning** — producing a "cleaned CSV" before owner confirms ambiguous decisions | Cases 01, 05 | "Do not modify data unless explicitly asked" + Questions-for-owner section |
| **Missing convention call-out** — knowing the right thing but not flagging the common trap | Case 02 | Explicit rule callouts in SKILL.md ("Convert vs localize") |
| **Auto-merging on weak evidence** — combining rows that look similar without a confidence threshold | Case 05 | "Resolution lives in a mapping layer; raw records are preserved" |
| **No regression artifact** — solving the immediate problem without leaving behind a check that a future change won't break it | Case 03 | "Check row counts before and after every join" |

## What's the same across both variants

- Knowledge of pandas / SQL / sklearn syntax: identical
- Speed of generating code: comparable
- Catching obvious sentinels, duplicates, and impossible dates: comparable
- Recognizing leakage patterns: comparable for this model

## Caveats

- **n = 1 per cell.** Single-shot, single-author, single-grader. Numbers indicate direction, not magnitude with a confidence interval.
- **The grader (Claude) is also the model under test.** Self-grading bias is real. Run `run.sh` against your own outputs and grade them yourself.
- **The rubric was written by the same author as the skills.** It rewards the format the skills produce. A more neutral rubric would test for *correctness of the eventual cleaned data*, not just response shape.
- **Cases are small (~12 rows).** They exercise the failure modes cleanly but real production datasets are messier in ways the model may not navigate as gracefully.
- **Only one model tested.** Anthropic Claude Opus 4.7 is at the top of the curve; weaker models likely show larger lift.

## How to make this rigorous

The runner script (`run.sh` for OAuth / `run.py` for API key) supports the easy parts:

- `SAMPLES=5 ./run.sh` for variance per cell
- `MODEL=sonnet ./run.sh` for cross-model comparison

What still requires human work:
- Blind grading by people who didn't write the skills
- A held-out case set written by someone other than the skill author
- Inter-rater agreement before trusting per-dimension scores
- Including weaker models in the mix — the lift curve probably bends sharply by capability

## Bottom-line interpretation

For Claude Opus 4.7, the skills produce a **+20 percentage point** lift on this 5-case slice. The lift is concentrated in tasks that reward discipline (entity resolution, profile-before-clean) and convention-awareness (localize-vs-convert). On tasks the model already does well (leakage audit, SCD2 joins), the skill is redundant or close to it.

That's a more modest claim than "skills transform the model" — and it's what the data actually shows.
