Stop before training. Several columns in this dataset leak the label. If you train on them, you will get cross-validation AUC ≈ 1.0 and brutal production performance.

## Prediction moment

You said "at the time of application". So the prediction moment is `application_date`. Anything created or updated after `application_date` is post-outcome and cannot be used as a feature.

What is known at prediction time:
- Applicant demographics
- Self-reported income
- Credit bureau pull (at application)
- Loan request details

What is not yet known:
- Whether underwriting will approve
- Anything about repayment behavior
- Anything about delinquency or collections

## Why those features are leaks (not just "highly predictive")

- `total_payments_made`: by construction, defaulters made fewer payments. Including this teaches the model "people who paid less defaulted more" — true but useless. Strongest possible leak.
- `latest_fico`: a FICO refreshed monthly *encodes* the default. Bureaus report delinquencies, which crater the score. Using `latest_fico` is using the future.
- `collections_calls`: by company policy, calls only happen on delinquent accounts. A nonzero count is essentially a label.
- `loan_status`: literally the outcome, with `default → is_default = 1` by construction. Trivial leak.
- `decision_date`: known 1–14 days post-application. Either drop, or replace with `decision_lag = decision_date - application_date` if you genuinely have it at scoring time (you probably don't, since the model is supposed to *inform* the decision).

```

annual_income
credit_score
requested_amount
dti_ratio

application_date  → keep for split + derive month/dow
```

## Other issues

**Right-censoring**
- The label `is_default` is "default within 12 months". Applications submitted in the last ~12 months may be artificially negative because the window has not closed yet. Filter the training set to applications with `application_date <= cutoff - 12 months`, where `cutoff` is the dataset snapshot date.

**Train/test split**
- A random 80/20 split will leak the future into the past. **Use a time-based split**: train on the older portion, test on the most recent portion (after the 12-month right-censoring filter).
- Check if any applicant has multiple loans. If yes, use group-aware splitting too — same applicant in train and test inflates AUC.

**Class imbalance**
- Default rates are typically 2–10%. Use class weights or a metric robust to imbalance (PR-AUC, recall@precision-target). Don't optimize plain accuracy.

**Preprocessing**
- Imputers, scalers, and encoders must be `.fit` on train only, `.transform` on test. Save the fitted preprocessor as part of the model artifact.

## Sanity check before training

After dropping the leaky features, run a quick model and look at the AUC. If it's still suspiciously high (> ~0.85 is unusual for default prediction with only those features), keep hunting — there's another leak you haven't found yet.

## Recommended actions

1. Remove `decision_date`, `loan_status`, `total_payments_made`, `latest_fico`, `collections_calls`.
2. Filter training rows to applications older than 12 months.
3. Time-based split.
4. If applicants can repeat, group-aware split.
5. Fit preprocessing on train only.
6. Document each removed feature and *why*, so future maintainers don't re-add them.
