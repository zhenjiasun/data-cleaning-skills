# Data Leakage Audit — Loan Default Prediction

## Prediction Moment Definition

| | |
|---|---|
| **Prediction moment** | At loan application submission (`application_date`) |
| **What is known** | Applicant demographics, self-reported financials, bureau pull at application |
| **What is unknown** | Whether the loan is approved, any repayment behavior, future credit score changes, collections activity |
| **Outcome window** | Default (90+ DPD) within 12 months of origination |

---

## Feature-by-Feature Audit

### ✅ Safe Features (known at application time)

| Feature | Rationale |
|---|---|
| `applicant_age` | Static at application |
| `annual_income` | Self-reported at application |
| `credit_score` | Bureau pull at application |
| `requested_amount` | Set by applicant |
| `dti_ratio` | Computed at application |
| `employment_years` | Reported at application |

### 🚨 Leaking Features — Must Remove

| Feature | Risk | Severity | Explanation |
|---|---|---|---|
| `decision_date` | **Post-prediction field** | 🟡 Medium | Occurs 1–14 days *after* application. The lag itself could encode underwriting complexity (harder cases take longer), which correlates with risk. Not available at prediction time. |
| `loan_status` | **Direct target leak** | 🔴 Critical | This *is* the outcome. `default` maps 1:1 to `is_default=1`. A model seeing this feature will learn to just read the answer. |
| `total_payments_made` | **Post-origination behavior** | 🔴 Critical | Payments happen months after application. Low values perfectly signal default. This alone would give you ~99% accuracy and 0% real-world value. |
| `latest_fico` | **Future-updated field** | 🔴 Critical | Refreshed monthly *after* origination. A FICO drop to 540 is a **consequence** of default, not a predictor of it. You already have `credit_score` (the point-in-time bureau pull). |
| `collections_calls` | **Post-default activity** | 🔴 Critical | Collections calls happen *because* the borrower defaulted. This is the most egregious leak — it's literally a downstream effect of the label. 14 calls → default is a tautology. |

> **If you trained with these features you'd see suspiciously perfect metrics (AUC > 0.99). That's not a good model — it's reading the answer sheet.**

## Recommended Train/Test Split

Your data has `application_date` — **use a time-based split, not random**.

```python
import pandas as pd

df = df.sort_values('application_date')

# Train: applications before cutoff
# Test:  applications after cutoff (simulates deploying to future applicants)
cutoff = '2023-05-01'

train = df[df['application_date'] < cutoff]
test  = df[df['application_date'] >= cutoff]
```

| Split detail | Recommendation |
|---|---|
| **Method** | Temporal split on `application_date` |
| **Why not random?** | Random splits let the model see future macro trends (rising unemployment, rate changes) that wouldn't be available in production. Temporal splits simulate real deployment. |
| **Cutoff heuristic** | ~70-80% of your date range for train, remainder for test |
| **Validation** | Use expanding-window or sliding-window time-series CV within the training set |

## Summary of Actions

1. **Remove** `loan_status`, `total_payments_made`, `latest_fico`, `collections_calls`, `decision_date` — all leak future information
2. **Keep** the 6 application-time features listed above
3. **Split** by `application_date`, not randomly
4. **Fit** all preprocessing on train only
5. **Expect realistic AUC** in the 0.65–0.78 range — if you see 0.95+, hunt for more leakage
6. **Consider engineering** additional point-in-time features (e.g., credit utilization at application, number of prior applications if you have that data) to recover predictive power lost by removing the leaked columns
