## The Interval Join

The key technique is joining where the order date falls within the address's validity window:

```sql
SELECT
    o.order_id,
    o.customer_id,
    c.address,
    o.order_date,
    o.amount
FROM orders o
JOIN customers c
  ON o.customer_id = c.customer_id
 AND o.order_date BETWEEN c.valid_from AND c.valid_to
ORDER BY o.customer_id, o.order_date;
```

## The "One Row per Customer" Problem

Here's the tension: you asked for **one row per customer**, but customers 101 and 103 each have orders tied to *two different addresses*. You have a few options:

### Option A — Use the most recent address (simple summary)

```sql
SELECT
    o.customer_id,
    curr.address AS current_address,
    SUM(o.amount) AS total_revenue
FROM orders o
JOIN customers curr
  ON o.customer_id = curr.customer_id
 AND curr.valid_to = '9999-12-31'        -- current row
GROUP BY o.customer_id, curr.address
ORDER BY o.customer_id;
```

| customer_id | current_address | total_revenue |
|-------------|-----------------|---------------|
| 101         | 18 Oak Ave      | 125.00        |
| 102         | 12 Pine Rd      | 120.00        |
| 103         | 22 Birch Ln     | 90.00         |
| 104         | 99 Cedar Ct     | 90.00         |

### Option B — One row per customer × address (preserves history)

```sql
SELECT
    o.customer_id,
    c.address,
    SUM(o.amount) AS total_revenue
FROM orders o
JOIN customers c
  ON o.customer_id = c.customer_id
 AND o.order_date BETWEEN c.valid_from AND c.valid_to
GROUP BY o.customer_id, c.address
ORDER BY o.customer_id, MIN(o.order_date);
```

| 101         | 18 Oak Ave  | 75.00         |

**Option A** is usually what downstream reports want (total revenue + where the customer lives *now*). **Option B** is better for auditing how revenue split across address changes. Which fits your use case?
