## ⚠️ The "one row per customer" conflict

You asked for **one row per customer** with total revenue *and* the address at the time of each order. These goals conflict for customers **101** and **103**, who placed orders under *different* addresses. You must choose a grain:

### Option A — One row per customer × address (preserves history)

```sql
SELECT
    o.customer_id,
    c.address,
    SUM(o.amount) AS total_revenue,
    COUNT(*)      AS order_count
FROM orders o
JOIN customers c
  ON o.customer_id = c.customer_id
 AND o.order_date >= c.valid_from
 AND o.order_date <= c.valid_to
GROUP BY o.customer_id, c.address;
```

### Option B — One row per customer, most-recent address only

```sql
WITH revenue AS (
    SELECT customer_id, SUM(amount) AS total_revenue
    FROM orders
    GROUP BY customer_id
),
current_address AS (
    SELECT customer_id, address
    FROM customers
    WHERE valid_to = '9999-12-31'   -- current row
)
SELECT
    r.customer_id,
    ca.address AS current_address,
    r.total_revenue
FROM revenue r
JOIN current_address ca
  ON r.customer_id = ca.customer_id;
```

## Recommendation

| If you need… | Use |
|---|---|
| Revenue attribution by address period | **Option A** (grain = customer × address) |
| A simple customer summary with their current address | **Option B** (aggregate first, then join to current dim row) |

The key principle: **aggregate to the desired grain *before* joining** whenever your fact and dimension have different cardinalities post-aggregation. That prevents accidental fanout and makes the intended grain explicit for future readers.
