## Intended grain

- One row per **order**, with the address that was on file at `order_date`.
- Cardinality: orders → customers should be **many-to-one** *given a point-in-time predicate*. Without that predicate, it becomes many-to-many.

## Correct join (SCD2 predicate)

```sql
SELECT
    o.customer_id,
    o.order_id,
    o.order_date,
    c.address AS address_at_order_time,
    o.amount
FROM orders o
LEFT JOIN customers c
  ON c.customer_id = o.customer_id
 AND o.order_date >= c.valid_from
 AND o.order_date <  c.valid_to    -- exclusive upper bound; '9999-12-31' = open-ended
;
```

## Aggregating to per-customer

If you want one row per customer with revenue and **most-recent address** (a different question — pick deliberately):

```sql
WITH per_order AS (
  SELECT o.customer_id, o.amount,
         c.address AS address_at_order_time
  FROM orders o
  LEFT JOIN customers c
    ON c.customer_id = o.customer_id
   AND o.order_date >= c.valid_from
   AND o.order_date <  c.valid_to
),
current_address AS (
  SELECT customer_id, address
  FROM customers
  WHERE valid_to = DATE '9999-12-31'
)
SELECT p.customer_id,
       SUM(p.amount) AS total_revenue,
       ca.address AS current_address
FROM per_order p
LEFT JOIN current_address ca USING (customer_id)
GROUP BY p.customer_id, ca.address
ORDER BY p.customer_id;
```

| 102 | 120 | 12 Pine Rd |

Sanity check: 125+120+90+90 = **$425** ✓.

## Pitfalls if you decide differently

- If you want the **most recent** address (not the as-of address), use `current_address` only — but be explicit, because customer 101's order 9001 was placed at `5 Main St`, not `18 Oak Ave`.
- If `valid_to` is **inclusive** in your shop's convention (rather than exclusive), use `<=`; check with the data owner. I assumed exclusive because `9999-12-31` is the canonical "open" sentinel.
- If you need to handle orders that pre-date the earliest address version (orphans), wrap with `COALESCE(c.address, '<unknown>')` and surface them in a `dq_orphans` view.
